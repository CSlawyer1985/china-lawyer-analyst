# 司法解释模块审核清单

---

## 审核流程

```
AI生成初稿 / 手动创建
    ↓
【第一层】结构验证（自动）
    ↓
【第二层】内容验证（AI辅助）
    ↓
【第三层】人工审核（最终把关）
    ↓
审核决策
```

---

## 第一层：结构验证（自动）

### 1.1 文件结构完整性

- [ ] 必备文件存在
  - [ ] README.md
  - [ ] index.md
  - [ ] metadata.json
- [ ] 可选文件（根据需要）
  - [ ] external-links.md
  - [ ] articles/ 目录
  - [ ] articles/article-{number}.md

### 1.2 JSON格式验证

- [ ] metadata.json 格式正确
  - [ ] JSON语法正确（可解析）
  - [ ] 必填字段完整
    - [ ] id
    - [ ] name
    - [ ] official_name
    - [ ] document_number
    - [ ] effective_date
    - [ ] article_count
    - [ ] keywords
    - [ ] priority_articles

### 1.3 命名规范验证

- [ ] 目录命名符合规范
  - [ ] 格式：`{领域}-{类型}-{年份}`
  - [ ] 示例：`contract-general-2023`、`security-law-2020`
- [ ] 文件命名符合规范
  - [ ] 条文详解：`article-{number}.md`
  - [ ] 小写字母、连字符分隔

### 1.4 链接验证

- [ ] 内部链接有效
  - [ ] README.md 中的相对链接可访问
  - [ ] index.md 中的相对链接可访问
- [ ] 外部链接格式正确（不验证可访问性）


## 第三层：人工审核（最终把关）

### 3.1 法律准确性审查

**审查要点**：
- [ ] 条文解读是否准确？
- [ ] 新旧法对比是否正确？
- [ ] 核心变化识别是否完整？
- [ ] 影响分析是否合理？

**审查方法**：
1. 对照官方文本逐条核对
2. 参考权威解读（威科先行、北大法宝）
3. 查阅相关案例（中国裁判文书网）

**常见问题**：
- ❌ 条文引用错误（编号、内容）
- ❌ 新旧法对比不准确
- ❌ 核心变化遗漏
- ❌ 影响分析不全面

### 3.2 深度分析能力审查

**核心竞争力验证**：
- [ ] ✅ 新旧对比分析
  - [ ] README.md 包含对比表格
  - [ ] 对比清晰准确
  - [ ] 突出显示重大变化

- [ ] ✅ 实务应用场景
  - [ ] articles/ 包含案例分析
  - [ ] 案情描述具体
  - [ ] 分析逻辑清晰
  - [ ] 结论明确

- [ ] ✅ 争议焦点讨论
  - [ ] articles/ 包含争议焦点
  - [ ] 焦点识别准确
  - [ ] 观点呈现全面
  - [ ] 有倾向性建议（如有）

**质量标准**：
- 优秀（⭐⭐⭐⭐⭐）：三项全部优秀
- 良好（⭐⭐⭐⭐）：三项全部达标
- 及格（⭐⭐⭐）：核心内容准确
- 不及格（⭐⭐）：需重新编写

### 3.3 实用性审查

**用户视角审查**：
- [ ] 快速查询场景
  - [ ] 能否通过 index.md 快速定位条文？
  - [ ] 主题描述是否清晰？

- [ ] 深度分析场景
  - [ ] 条文详解是否足够深入？
  - [ ] 实务案例是否有参考价值？
  - [ ] 争议焦点是否有启发？

- [ ] 外部检索场景
  - [ ] external-links.md 是否提供足够的数据源？
  - [ ] 使用指南是否清晰？

### 3.4 可维护性审查

**长期维护考虑**：
- [ ] 外部链接有效性
  - [ ] 官方链接稳定
  - [ ] 备用链接（如有）

- [ ] 元数据准确性
  - [ ] metadata.json 字段完整
  - [ ] 关键词合理
  - [ ] 优先级条文准确

- [ ] 版本管理
  - [ ] 版本号合理
  - [ ] 更新日期准确
  - [ ] 更新日志完整

## 审核决策

### 决策矩阵

| 评估维度 | 通过 | 需修改 | 不通过 |
|---------|------|--------|--------|
| 结构验证 | ✅ 全部通过 | ⚠️ 部分缺失 | ❌ 严重缺失 |
| 内容验证 | ⭐⭐⭐⭐ 及格以上 | ⭐⭐ 需改进 | ⭐ 不及格 |
| 深度分析 | ⭐⭐⭐⭐ 三项达标 | ⭐⭐⭐ 部分达标 | ⭐⭐ 不足 |
| 实用性 | ⭐⭐⭐⭐ 良好 | ⭐⭐⭐ 一般 | ⭐⭐ 较差 |
| 可维护性 | ✅ 优秀 | ⚠️ 需优化 | ❌ 不足 |

### 决策标准

**✅ 通过（合并到主分支）**
- 结构验证：✅ 全部通过
- 内容验证：⭐⭐⭐⭐ 及格以上
- 深度分析：⭐⭐⭐⭐ 三项达标
- 实用性：⭐⭐⭐⭐ 良好
- 可维护性：✅ 优秀

**⚠️ 需修改（返回修改后重新审核）**
- 结构验证：⚠️ 部分缺失（可快速修复）
- 内容验证：⭐⭐⭐ 一般
- 深度分析：⭐⭐⭐ 部分达标
- 实用性：⭐⭐⭐ 一般
- 可维护性：⚠️ 需优化

**❌ 不通过（人工重写）**
- 结构验证：❌ 严重缺失
- 内容验证：⭐ 不及格
- 深度分析：⭐⭐ 不足
- 实用性：⭐⭐ 较差
- 可维护性：❌ 不足

## 常见问题示例

### 问题1：条文解读不准确

**表现**：
- 条文内容引用错误
- 核心要点遗漏
- 与官方解读不符

**处理**：
- ❌ 不通过
- 要求：对照官方文本重新编写

### 问题2：新旧法对比不完整

**表现**：
- 仅列出新法，未对比旧法
- 对比不准确
- 变化影响未说明

**处理**：
- ⚠️ 需修改
- 要求：补充旧法内容和对比分析

### 问题3：深度分析不足

**表现**：
- 无实务案例
- 无争议焦点讨论
- 分析过于浅显

**处理**：
- ⚠️ 需修改
- 要求：补充实务案例和争议焦点

### 问题4：Token超标

**表现**：
- README.md > 1,000 tokens
- index.md > 500 tokens
- article-X.md > 300 tokens

**处理**：
- ⚠️ 需修改
- 要求：精简内容，删除冗余信息
